9.2. Filesystems provided to the guest¶

Charliecloud guests receive both metadata about the job and data of the user’s choice using 9p virtio filesystem passthrough. In standard images, these are mounted automatically under /ch. The mount tag for each is the same as the final directory given below, which can be used to identify the filesystem in other guests (e.g.: /ch/meta gets mount tag meta).

9.2.1. Metadata¶

Metadata is available under /ch/meta. The following files are provided. All guests in the job get the same files.

Files containing records have one record per line; fields are separated by whitespace. Record order is arbitrary.

  • commit: If present, the root filesystem of least one virtual machine in the cluster will be committed. If absent, all changes will be discarded. The point is to let jobs know if they can safely mess with the root filesystem.
  • guest-macs: Two fields: integer guest ID followed by MAC address.
  • guest-ips: Three fields: integer guest ID, IP address of guest, IP address of corresponding host. Note that this file may be incomplete and may list guests that are unreachable.
  • host-userdata: Python pickle file describing the user and groups of the invoking host user.
  • interactive: Indicates that the boot scripts have requested an interactive job despite providing a job script. If present, the standard images do not shut down after job.sh finishes.
  • job.sh: Job shell script provided to guest boot script. If present, the standard images run this script after boot. (Note: If job.sh is not present, standard images will stay up after boot regardless of interactive.) (FIXME: Document that sudo /etc/rc.local will re-run the job as if after boot, and document that doing so won’t shut down regardless of above.)

9.2.2. Charliecloud resources¶

/ch/opt contains guest resources that come with Charliecloud, such as scripts to run the job. Standard images require these resources in order to do boot as documented here, but they will come up in a minimal configuration if they are absent.

9.2.3. Persistent storage¶

Directories selected by the user (using the -d switch of runguest or runguests) are offered to the guests under /ch/data[1-4]. (There’s no inherent limit, but 4 is what we implemented in the standard images.)

9.2.4. Temporary storage¶

A block storage device is offered to each guest for temporary file storage; it is wiped after the guest shuts down.

Standard guests create an ext4 filesystem with no journal and mount it under /ch/tmp, option noatime.
